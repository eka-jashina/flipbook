# Деплой Flipbook на Amvera Cloud

Подробная пошаговая инструкция: от нуля до работающего production-сайта.

## Что получится в итоге

```
┌─────────────────────────────────────────────────────────┐
│                    Amvera Cloud                         │
│                                                         │
│  ┌───────────────────────┐  ┌───────────────────────┐  │
│  │   flipbook (Docker)   │  │   flipbook-db          │  │
│  │                       │  │   (PostgreSQL 17)      │  │
│  │  Express API (:4000)  │──│                        │  │
│  │  + фронтенд (SPA)    │  │   Управляемая БД       │  │
│  │  + Prisma ORM         │  │   Бэкапы автоматом     │  │
│  └───────────┬───────────┘  └───────────────────────┘  │
│              │                                          │
│  Бесплатный домен: https://flipbook-ЛОГИН.amvera.io    │
│  HTTPS-сертификат выпускается автоматически              │
└──────────────┬──────────────────────────────────────────┘
               │
       ┌───────┴───────┐
       │  Cloud.ru S3  │
       │  (15 ГБ free) │
       │  Файлы, фонты │
       │  аватарки      │
       └───────────────┘
```

- **Сервер** (Express + фронтенд в одном Docker-контейнере) — Amvera Cloud
- **База данных** (PostgreSQL 17) — управляемая БД на Amvera Cloud
- **Хранилище файлов** (S3) — Cloud.ru Object Storage (15 ГБ бесплатно навсегда)
- **Домен + HTTPS** — бесплатно от Amvera (или свой домен)

## Стоимость

| Сервис | Стоимость | Что даёт |
|--------|-----------|----------|
| Amvera — сервер | ~290 руб/мес (тариф «Начальный») | 0.25 CPU, 512 МБ RAM |
| Amvera — БД | ~290 руб/мес (тариф «Начальный») | PostgreSQL 17, 0.25 CPU, 512 МБ |
| Cloud.ru S3 | Бесплатно | 15 ГБ хранилище навсегда |
| **Итого** | **~580 руб/мес** | |

> При регистрации на Amvera дают **111 руб. бонусом** — хватит на ~3 недели работы.
> Есть «Пробный» тариф (дешевле, 0.1 CPU / 100 МБ RAM) — подходит для тестирования.

## Часть 1. Cloud.ru — S3-хранилище для файлов

Начинаем с Cloud.ru, потому что ключи S3 понадобятся при настройке сервера на Amvera.

### Зачем нужен S3?

Flipbook хранит в S3 загруженные пользователями файлы:
- Обложки книг
- Аватарки пользователей
- Кастомные шрифты (woff2, ttf, otf)
- Фоновые изображения глав
- Звуковые файлы (эмбиент, перелистывание)

Без S3 загрузка файлов не будет работать — сервер вернёт ошибку.

### Шаг 1.1. Регистрация на Cloud.ru

1. Открой https://cloud.ru/offers/free-tier
2. Нажми **«Попробовать бесплатно»** (или «Регистрация»)
3. Введи email и придумай пароль
4. Подтверди email — придёт письмо, кликни ссылку в нём
5. Заполни данные (имя, телефон)
6. После подтверждения попадёшь в консоль управления Cloud.ru

### Шаг 1.2. Создай проект

1. В левом меню консоли Cloud.ru выбери **«Все проекты»**
2. Если проектов нет — нажми **«Создать проект»**
3. Назови его `flipbook` (или как угодно — это просто группировка ресурсов)
4. Нажми **«Создать»**
5. Кликни на созданный проект, чтобы войти в него

> Если проект создался автоматически при регистрации — используй его.

### Шаг 1.3. Создай бакет (хранилище)

Бакет — это «папка» в облачном хранилище, куда будут загружаться файлы.

1. В левом меню найди **«Object Storage»** (или через поиск сервисов наверху)
2. Нажми **«Создать бакет»**
3. Заполни:
   - **Имя бакета:** `flipbook-uploads`
     - Только строчные буквы, цифры и дефисы
     - Имя должно быть глобально уникальным — если занято, добавь суффикс (например, `flipbook-uploads-123`)
   - **Класс хранения:** Стандартный
   - Остальное оставь по умолчанию
4. Нажми **«Создать»**

### Шаг 1.4. Включи публичный доступ на чтение

Это нужно, чтобы браузер мог загружать картинки, шрифты и звуки напрямую из хранилища. Без этого пользователи не увидят обложки книг и кастомные шрифты.

1. Открой созданный бакет `flipbook-uploads`
2. Перейди в **«Настройки»** (или «Settings»)
3. Найди секцию **«Публичный доступ»** (Public access)
4. Включи **«Чтение объектов»** (GET) — это разрешит только скачивание файлов, загрузка по-прежнему будет только через API с авторизацией
5. Сохрани изменения

> **Безопасность:** Публичный доступ даёт только чтение. Загрузка файлов идёт через сервер с проверкой авторизации. Удалять файлы напрямую из бакета нельзя — только через API.

### Шаг 1.5. Создай API-ключи

API-ключи нужны серверу, чтобы загружать файлы в хранилище от имени приложения.

1. Нажми на свои инициалы в правом верхнем углу → **«Мой профиль»**
2. Перейди в раздел **«Ключи доступа»** (или API Keys)
3. Нажми **«Создать ключ»** (или «Сгенерировать»)
4. Появятся два значения:
   ```
   Key ID:     AKIAIOSFODNN7EXAMPLE        ← это S3_ACCESS_KEY
   Key Secret: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY  ← это S3_SECRET_KEY
   ```
5. **ОБЯЗАТЕЛЬНО СКОПИРУЙ ОБА ЗНАЧЕНИЯ СЕЙЧАС!** Key Secret показывают только один раз — потом его нельзя будет посмотреть, только пересоздать
6. Сохрани в надёжное место (менеджер паролей, заметки)

### Шаг 1.6. Запиши итоговые данные S3

Теперь у тебя есть всё для подключения S3. Запиши все значения — они понадобятся в шаге 2.6:

```
S3_ENDPOINT=https://s3.cloud.ru
S3_BUCKET=flipbook-uploads
S3_REGION=ru-1
S3_ACCESS_KEY=<Key ID из шага 1.5>
S3_SECRET_KEY=<Key Secret из шага 1.5>
S3_FORCE_PATH_STYLE=true
S3_PUBLIC_URL=https://s3.cloud.ru/flipbook-uploads
```

> **Примечание:** Точный endpoint и регион могут отличаться — проверь в документации Cloud.ru или в настройках бакета. Публичный URL бакета тоже будет виден в настройках.

> **Что значит S3_FORCE_PATH_STYLE?** Cloud.ru использует path-style URL'ы (`https://s3.cloud.ru/bucket-name/file`) вместо virtual-hosted (`https://bucket-name.s3.cloud.ru/file`). Поэтому эта опция должна быть `true`.

## Часть 2. Amvera Cloud — сервер и база данных

### Шаг 2.1. Регистрация на Amvera

1. Открой https://amvera.ru
2. Нажми **«Регистрация»** (правый верхний угол)
3. Введи email, придумай пароль, введи имя пользователя (логин — латиницей)
4. Подтверди номер телефона (придёт SMS-код)
5. После регистрации на балансе появится **111 руб.** стартовый бонус
6. Ты попадёшь на страницу **«Проекты»**

> **Запомни свой логин** — он будет частью доменного имени: `https://flipbook-ЛОГИН.amvera.io`

### Шаг 2.2. Создай базу данных PostgreSQL

1. На странице проектов нажми **«Создать»**
2. Выбери тип: **«PostgreSQL»** (не «Приложение»!)
3. Заполни:
   - **Название проекта:** `flipbook-db` (латиницей, без пробелов)
   - **Тариф:**
     - **Начальный** (0.25 CPU, 512 МБ RAM) — рекомендуется для стабильной работы
     - **Пробный** (0.1 CPU, 100 МБ RAM) — для экономии на начальном этапе
4. Нажми **«Далее»**
5. Заполни параметры базы данных:
   - **Имя базы данных:** `flipbook`
   - **Имя пользователя:** `flipbook`
   - **Пароль:** придумай надёжный пароль и **обязательно запиши!**
     - Пример: `MyDbPass123!` (минимум 8 символов, буквы + цифры)
     - Этот пароль нельзя будет потом посмотреть — только сбросить
6. Нажми **«Завершить»**
7. Подожди, пока статус станет **«PostgreSQL запущен»** — обычно 1–3 минуты

### Шаг 2.3. Найди хост базы данных

Хост — это внутренний адрес базы данных внутри сети Amvera. Сервер будет подключаться к базе по этому адресу.

1. Открой проект `flipbook-db`
2. Перейди на вкладку **«Инфо»**
3. Найди **внутреннее доменное имя** — оно выглядит примерно так:
   ```
   amvera-ТВОЙЛОГИН-cnpg-flipbook-db-rw
   ```
4. **Скопируй его** — это хост для строки подключения

> Формат может немного отличаться. Главное — скопируй то, что показано на вкладке «Инфо».

### Шаг 2.4. Собери DATABASE_URL

Из данных шагов 2.2 и 2.3 собери строку подключения к базе. Это стандартный формат PostgreSQL connection string:

```
postgresql://ПОЛЬЗОВАТЕЛЬ:ПАРОЛЬ@ХОСТ:5432/БАЗА?sslmode=disable&connection_limit=20&pool_timeout=15
```

Подставь свои значения:

```
postgresql://flipbook:MyDbPass123!@amvera-ТВОЙЛОГИН-cnpg-flipbook-db-rw:5432/flipbook?sslmode=disable&connection_limit=20&pool_timeout=15
```

> **Что означают параметры:**
> - `sslmode=disable` — обязательно для подключения внутри Amvera (между проектами, трафик и так изолирован)
> - `connection_limit=20` — максимум 20 одновременных соединений в пуле Prisma (оптимально для тарифа «Начальный»)
> - `pool_timeout=15` — таймаут ожидания свободного соединения в секундах

> **Ошибки в DATABASE_URL — самая частая причина проблем при деплое.** Перепроверь каждую часть: пользователя, пароль (без лишних пробелов!), хост (скопирован полностью), порт 5432, имя базы.

### Шаг 2.5. Создай проект для сервера

1. Вернись на страницу **«Проекты»**
2. Нажми **«Создать»**
3. Выбери тип: **«Приложение»** (не «PostgreSQL»!)
4. Заполни:
   - **Название проекта:** `flipbook` (латиницей)
   - **Тариф:**
     - **Начальный** (0.25 CPU, 512 МБ RAM) — рекомендуется
     - **Пробный** — для теста
5. Метод загрузки: **«Через Git»** — Amvera даёт собственный Git-репозиторий, при push туда проект автоматически пересобирается
6. Нажми **«Завершить»**

### Шаг 2.6. Добавь переменные окружения

**Это самый важный шаг! Без правильных переменных сервер не запустится.**

Сервер использует Zod-валидацию переменных (`server/src/config.ts`). При запуске в production-режиме обязательны: `DATABASE_URL`, `SESSION_SECRET`, `CSRF_SECRET`, `S3_ACCESS_KEY`, `S3_SECRET_KEY`, `S3_PUBLIC_URL`, `CORS_ORIGIN`, `APP_URL`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`.

1. Открой проект `flipbook`
2. Перейди на вкладку **«Переменные»**
3. Для каждой переменной: нажми **«Создать переменную»**, выбери тип (переменная или секрет), заполни имя и значение

#### Обычные переменные (тип: «переменная»)

Эти значения не секретные — их можно показывать в логах.

| Имя | Значение | Зачем |
|-----|----------|-------|
| `NODE_ENV` | `production` | Включает production-режим: минификация логов, secure cookies, строгая валидация |
| `PORT` | `4000` | Порт, на котором слушает Express (должен совпадать с `containerPort` в `amvera.yml`) |
| `SESSION_SECURE` | `true` | Cookies передаются только по HTTPS (Amvera даёт HTTPS автоматически) |
| `S3_BUCKET` | `flipbook-uploads` | Имя бакета из шага 1.3 |
| `S3_REGION` | `ru-1` | Регион Cloud.ru (проверь в настройках бакета) |
| `S3_FORCE_PATH_STYLE` | `true` | Cloud.ru использует path-style URL'ы (см. шаг 1.6) |
| `RUN_MIGRATIONS` | `true` | Автоматически применять миграции Prisma при старте контейнера |

> **Про RUN_MIGRATIONS:** При первом деплое `true` обязательно — без этого таблицы в БД не создадутся и сервер упадёт. В дальнейшем можно оставить `true` — миграции идемпотентны (повторный запуск ничего не сломает) и занимают ~1 секунду при отсутствии изменений.

#### Секреты (тип: «секрет»)

Конфиденциальные данные. Amvera хранит их зашифрованно и не показывает в логах.

| Имя | Значение | Зачем |
|-----|----------|-------|
| `DATABASE_URL` | `postgresql://flipbook:...` (из шага 2.4) | Строка подключения к PostgreSQL |
| `SESSION_SECRET` | *(случайная строка ≥32 символа)* | Ключ шифрования сессий (express-session) |
| `CSRF_SECRET` | *(другая случайная строка ≥32 символа)* | Ключ для CSRF-токенов (csrf-csrf) |
| `S3_ENDPOINT` | `https://s3.cloud.ru` | Адрес S3-совместимого хранилища |
| `S3_ACCESS_KEY` | *(Key ID из шага 1.5)* | Ключ доступа к S3 |
| `S3_SECRET_KEY` | *(Key Secret из шага 1.5)* | Секретный ключ S3 |
| `S3_PUBLIC_URL` | `https://s3.cloud.ru/flipbook-uploads` | Публичный URL бакета (браузер загружает файлы оттуда) |

> **ВАЖНО:** `SESSION_SECRET` и `CSRF_SECRET` должны быть **разными** строками. Если они совпадают, это ослабляет безопасность.

> **Как сгенерировать случайную строку:**
> ```bash
> # Node.js (рекомендуется)
> node -e "console.log(require('crypto').randomBytes(48).toString('base64url'))"
>
> # Python
> python3 -c "import secrets; print(secrets.token_urlsafe(48))"
>
> # OpenSSL
> openssl rand -base64 48
> ```
> Выполни команду дважды — одну строку для `SESSION_SECRET`, другую для `CSRF_SECRET`.

#### Переменные, которые добавляются ПОСЛЕ получения домена

Эти переменные зависят от URL'а приложения, который ты узнаешь только на шаге 2.9. **Добавь их позже:**

| Имя | Значение | Зачем |
|-----|----------|-------|
| `CORS_ORIGIN` | `https://flipbook-ЛОГИН.amvera.io` | Разрешённые источники для CORS (браузер блокирует API-запросы без этого) |
| `APP_URL` | `https://flipbook-ЛОГИН.amvera.io` | Базовый URL приложения (для ссылок в email, SEO) |
| `GOOGLE_CLIENT_ID` | *(из Google Cloud Console)* | ID для Google OAuth (см. раздел «Google OAuth») |
| `GOOGLE_CLIENT_SECRET` | *(из Google Cloud Console)* | Секрет для Google OAuth |
| `GOOGLE_CALLBACK_URL` | `https://flipbook-ЛОГИН.amvera.io/api/auth/google/callback` | URL обратного вызова Google OAuth |

> **Без CORS_ORIGIN и APP_URL** сервер запустится, но: фронтенд не сможет делать API-запросы (CORS ошибки в браузере), авторизация не будет работать.

> **Google OAuth обязателен в production.** Сервер валидирует, что `GOOGLE_CLIENT_ID` и `GOOGLE_CLIENT_SECRET` заданы и не равны `placeholder`. Если Google OAuth пока не нужен — см. раздел «Google OAuth» ниже, где описано, как получить эти ключи.

### Шаг 2.7. Подключи Git-репозиторий

Amvera предоставляет собственный Git-сервер. При push в него автоматически запускается сборка Docker-образа и деплой.

Открой терминал в папке проекта Flipbook:

```bash
# 1. Добавь Amvera как дополнительный remote
# URL виден на вкладке «Репозиторий» в проекте Amvera
git remote add amvera https://git.amvera.ru/ТВОЙЛОГИН/flipbook

# 2. Запуши код в Amvera
git push amvera main
```

> **При первом пуше** терминал спросит логин и пароль — введи данные от аккаунта Amvera (логин и пароль, которые ты создал при регистрации).

> **Если ветка называется `master`**, а не `main`:
> ```bash
> git push amvera master:main
> ```
> Amvera ожидает ветку `main`. Команда `master:main` пушит локальную ветку `master` в удалённую `main`.

> **Если ошибка `failed to push some refs`:**
> ```bash
> git pull amvera main --allow-unrelated-histories
> # Разреши конфликты, если есть, и запуши снова:
> git push amvera main
> ```

### Шаг 2.8. Наблюдай за сборкой

1. В интерфейсе Amvera открой проект `flipbook`
2. Перейди на вкладку **«Лог сборки»**
3. Ты увидишь, как Docker собирает образ по стадиям:
   - **Stage 1 (frontend-builder):** установка npm-зависимостей, сборка фронтенда через Vite
   - **Stage 2 (backend-builder):** установка серверных зависимостей, генерация Prisma-клиента, компиляция TypeScript
   - **Stage 3 (production):** сборка финального образа с dumb-init, копирование артефактов
4. Сборка занимает **3–7 минут** (зависит от тарифа и кэша)
5. После сборки перейди на **«Лог приложения»** — увидишь запуск сервера:
   ```
   prisma migrate deploy  ← миграции (если RUN_MIGRATIONS=true)
   Server listening on port 4000  ← сервер запустился
   ```
6. Жди статуса **«Приложение запущено»** на вкладке «Инфо»

> **Если в логах ошибка** — 99% проблем связаны с переменными окружения:
> - `Error: DATABASE_URL must start with postgresql://` → проверь формат DATABASE_URL
> - `Error: S3_ACCESS_KEY is required in production` → забыл добавить S3-ключи
> - `Error: CORS_ORIGIN is required in production` → добавь CORS_ORIGIN (шаг 2.10)
> - `Connection refused` → база данных ещё не запустилась (подожди 1–2 минуты) или неправильный хост в DATABASE_URL

### Шаг 2.9. Включи домен

1. Перейди на вкладку **«Настройки»** проекта `flipbook`
2. Найди секцию **«Доменное имя»**
3. Включи тумблер **«Бесплатный домен»**
4. Amvera покажет URL, например:
   ```
   https://flipbook-ТВОЙЛОГИН.amvera.io
   ```
5. **Скопируй этот URL** — он понадобится в следующем шаге

> HTTPS-сертификат выпускается автоматически. Первый запрос может занять несколько секунд, пока сертификат генерируется.

### Шаг 2.10. Добавь CORS_ORIGIN и APP_URL

Теперь, когда домен известен, добавь оставшиеся переменные:

1. Перейди на вкладку **«Переменные»**
2. Добавь:

| Имя | Значение |
|-----|----------|
| `CORS_ORIGIN` | `https://flipbook-ЛОГИН.amvera.io` |
| `APP_URL` | `https://flipbook-ЛОГИН.amvera.io` |
| `GOOGLE_CALLBACK_URL` | `https://flipbook-ЛОГИН.amvera.io/api/auth/google/callback` |

3. Amvera автоматически перезапустит контейнер с новыми переменными

> **Несколько доменов в CORS_ORIGIN:** если используешь и Amvera-домен, и свой — перечисли через запятую без пробелов:
> ```
> https://flipbook-ЛОГИН.amvera.io,https://myflipbook.com
> ```

## Часть 3. Проверка

### 3.1. Проверь здоровье сервера

Открой в браузере:
```
https://flipbook-ЛОГИН.amvera.io/api/health
```

Если всё настроено правильно, увидишь:
```json
{"status": "ok", "checks": {"database": "ok", "storage": "ok"}}
```

**Диагностика проблем:**

| Ответ | Причина | Решение |
|-------|---------|---------|
| `{"database": "error"}` | Неправильный DATABASE_URL | Проверь хост, пароль, имя базы. Убедись, что БД запущена |
| `{"storage": "error"}` | Неправильные S3-ключи или endpoint | Проверь S3_ENDPOINT, S3_ACCESS_KEY, S3_SECRET_KEY |
| Страница не открывается | Контейнер не запустился | Смотри «Лог приложения» в Amvera |
| 502 Bad Gateway | Контейнер запускается или упал | Подожди 1–2 минуты, проверь логи |

### 3.2. Открой сайт

Перейди на:
```
https://flipbook-ЛОГИН.amvera.io
```

Должна открыться лендинг-страница Flipbook. Попробуй:
1. Зарегистрироваться (email + пароль)
2. Создать книгу (загрузить txt/docx файл или добавить главы вручную)
3. Открыть книгу на чтение — страницы должны перелистываться

## Часть 4. Обновление

### Обычное обновление

После любых изменений в коде:

```bash
git add .
git commit -m "описание изменений"
git push amvera main
```

Amvera автоматически:
1. Обнаружит push
2. Пересоберёт Docker-образ (3–7 минут, кэшированные слои ускоряют)
3. Применит миграции БД (если `RUN_MIGRATIONS=true` и есть новые)
4. Перезапустит контейнер с новым образом

### Обновление с миграциями базы данных

Если ты изменил `server/prisma/schema.prisma` (добавил/удалил/изменил модели):

```bash
# 1. Создай миграцию локально
cd server
npx prisma migrate dev --name описание_изменения

# 2. Закоммить и запуши
cd ..
git add .
git commit -m "add migration: описание"
git push amvera main
```

При запуске контейнера `prisma migrate deploy` автоматически применит новые миграции.

> **Миграции безопасны:** `prisma migrate deploy` применяет только новые (ещё не применённые) миграции. Уже применённые пропускаются. Данные не теряются.

### Zero-downtime? Нет

При каждом деплое будет кратковременный даунтайм (~10–30 секунд), пока новый контейнер запускается. Это нормально для тарифа «Начальный» с одним инстансом.

## Часть 5. Google OAuth (обязательно для production)

Сервер в production-режиме требует `GOOGLE_CLIENT_ID` и `GOOGLE_CLIENT_SECRET` (не равные `placeholder`). Вот как их получить:

### Шаг 5.1. Создай проект в Google Cloud

1. Открой https://console.cloud.google.com/
2. Нажми **«Select a project»** → **«New Project»**
3. Назови проект (например, `Flipbook`) и создай

### Шаг 5.2. Настрой OAuth consent screen

1. В левом меню: **APIs & Services** → **OAuth consent screen**
2. Выбери **External** (для доступа любым Google-аккаунтам)
3. Заполни:
   - **App name:** Flipbook
   - **User support email:** твой email
   - **Developer contact email:** твой email
4. Scopes: добавь `email` и `profile`
5. Сохрани

### Шаг 5.3. Создай OAuth 2.0 credentials

1. **APIs & Services** → **Credentials** → **Create Credentials** → **OAuth 2.0 Client ID**
2. **Application type:** Web application
3. **Name:** Flipbook (или любое)
4. **Authorized redirect URIs** — добавь:
   ```
   https://flipbook-ЛОГИН.amvera.io/api/auth/google/callback
   ```
5. Нажми **Create**
6. Скопируй **Client ID** и **Client Secret**

### Шаг 5.4. Добавь переменные в Amvera

В проекте `flipbook` на вкладке «Переменные» добавь:

| Имя | Значение |
|-----|----------|
| `GOOGLE_CLIENT_ID` | *(Client ID из Google)* |
| `GOOGLE_CLIENT_SECRET` | *(Client Secret из Google)* |
| `GOOGLE_CALLBACK_URL` | `https://flipbook-ЛОГИН.amvera.io/api/auth/google/callback` |

Amvera перезапустит контейнер. После этого на странице входа появится кнопка **«Войти через Google»**.

## Часть 6. Свой домен (опционально)

Если хочешь использовать свой домен вместо `*.amvera.io`:

1. В настройках проекта Amvera нажми **«Добавить домен»**
2. Введи свой домен (например, `flipbook.example.com`)
3. У регистратора домена (где ты купил домен) добавь DNS-запись:
   - **Тип:** CNAME
   - **Имя:** `flipbook` (или `@` для корневого домена)
   - **Значение:** `flipbook-ЛОГИН.amvera.io`
4. Подожди 5–30 минут (DNS-обновление)
5. Amvera автоматически выпустит SSL-сертификат для нового домена
6. Обнови переменные в Amvera:
   - `CORS_ORIGIN` = `https://flipbook.example.com`
   - `APP_URL` = `https://flipbook.example.com`
   - `GOOGLE_CALLBACK_URL` = `https://flipbook.example.com/api/auth/google/callback`
7. В Google Cloud Console добавь новый redirect URI в OAuth credentials

## Часть 7. Мониторинг и обслуживание

### Логи

- **Лог сборки** — вкладка «Лог сборки» в проекте Amvera (Docker build output)
- **Лог приложения** — вкладка «Лог приложения» (stdout/stderr сервера)
- Сервер использует Pino logger — логи в JSON-формате, удобно для поиска ошибок

### Статус

- **Вкладка «Инфо»** — текущее состояние контейнера, использование ресурсов
- **`/api/health`** — программная проверка здоровья (БД + S3)

### Sentry (опционально)

Для мониторинга ошибок в production:

1. Зарегистрируйся на https://sentry.io (бесплатный план до 5000 ошибок/мес)
2. Создай проект (Node.js)
3. Добавь переменную в Amvera:
   - `SENTRY_DSN` = *(DSN из Sentry)*
4. Сервер автоматически начнёт отправлять ошибки в Sentry

### Резервные копии БД

Amvera автоматически бэкапит управляемые PostgreSQL-базы. Для ручного бэкапа можно подключиться к БД через `pg_dump` (потребуется внешний доступ к БД — настраивается в Amvera).

## Опциональные переменные окружения

Эти переменные имеют разумные значения по умолчанию и нужны только для тонкой настройки:

| Имя | Значение по умолчанию | Описание |
|-----|----------------------|----------|
| `RATE_LIMIT_WINDOW` | `60000` (60 сек) | Окно ограничения запросов (мс). За это время один IP может сделать не более `RATE_LIMIT_MAX` запросов |
| `RATE_LIMIT_MAX` | `100` | Максимум запросов на одно окно на один IP-адрес |
| `SESSION_MAX_AGE` | `604800000` (~7 дней) | Время жизни сессии в миллисекундах. После этого срока пользователь должен войти снова |
| `SENTRY_DSN` | *(не задан)* | DSN для мониторинга ошибок в Sentry. Если не задан — Sentry не используется |

## Архитектура Docker-образа

Проект использует multi-stage Docker build (`Dockerfile` в корне проекта):

```
Stage 1: frontend-builder (node:22-alpine)
├── npm ci (фронтенд-зависимости)
└── vite build → dist/

Stage 2: backend-builder (node:22-alpine)
├── npm ci (серверные зависимости)
├── prisma generate (генерация Prisma-клиента)
└── tsc → dist/ (компиляция TypeScript)

Stage 3: production (node:22-alpine)
├── dumb-init (корректная обработка сигналов)
├── npm ci --omit=dev (только production-зависимости)
├── Prisma schema + клиент (из stage 2)
├── Скомпилированный сервер (из stage 2)
├── Собранный фронтенд (из stage 1) → /app/client/
├── Healthcheck: GET /api/health (каждые 30 сек)
└── CMD: [prisma migrate deploy] + node dist/index.js
```

Конфигурация Amvera (`amvera.yml`):
- `containerPort: 4000` — порт контейнера
- `persistenceMount: /data` — точка монтирования для персистентных данных
